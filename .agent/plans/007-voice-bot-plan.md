# 007 — Voice-Enabled Data Acquisition (Chatbot Implementation Plan)

Breaking the voice AI implementation into two distinct phases ensures we get a working feature quickly and avoid getting bogged down in complex architecture.

---

## Phase 1: The Bare Minimum (MVP)
*Goal: Get a working voice bot that listens, talks, extracts a JSON payload, and looks cool for a demonstration.*

**1. Platform Selection**
- Create a free account on **Vapi.ai** (easiest web integration for our React frontend).

**2. The System Prompt (Dashboard Setup)**
- Within the Vapi dashboard, write a clear system prompt:
  > *"You are a triage assistant. Your job is to collect the patient's age, chief complaint, heart rate, and blood pressure. Ask questions one at a time. Be polite and concise."*

**3. Define the Tool (Data Schema)**
- In Vapi, define a "Custom Tool" containing a simplified version of our Triage schema (e.g., just `age`, `chief_complaint`, `heart_rate`).
- Instruct the AI to trigger this tool once it has collected all the required answers. 

**4. Frontend Integration**
- Add the Vapi React SDK to our frontend (`npm install @vapi-ai/web`).
- Create a simple "🎙️ Start Voice Triage" button alongside our existing web form.
- When the call ends, capture the JSON output generated by the AI and automatically populate the frontend triage form with those values so the user can review them before clicking "Submit to ML Model".

*(Note: In Phase 1, there is no LangChain, no complex backend webhooks, and no rigorous validation. We rely entirely on the platform's out-of-the-box LLM to do a "good enough" job finding the variables.)*

---

## Phase 2: The "ML-Grade" Polish (Advanced Improvements)
*Goal: Turn the voice bot into a rigorous data-acquisition pipeline designed for machine learning readiness, suitable for presenting to an ML professor.*

**1. Real-Time Backend Validation (Guardrails)**
- Connect the Vapi custom tool to a live webhook endpoint on our FastAPI backend.
- When the AI extracts a value (e.g., `heart_rate: 300`), the backend intercepts it, runs it against our clinical thresholds, and returns an error response instructing the AI to re-ask the question.

**2. Dual-Model Dynamic Routing (Integration with Model v1.0.2-c)**
- If the AI fails to extract the 25 complex medical history features from the user (because the user hung up early or didn't know), the backend automatically notices the missing fields.
- The backend seamlessly reroutes the payload to our Fallback Model (`v1.0.2-c`) instead of crashing the primary model (`v1.0.2`).

**3. Chief Complaint Normalization (Domain Shift Fix)**
- The v1.0.2-b CC System Classifier was trained on terse, nurse-typed phrases (e.g., `"chest pain radiating to left arm"`).
- Conversational voice input produces verbose, informal text (e.g., `"my chest has been hurting and it goes into my left arm"`), which shifts the BioBERT → PCA coordinates away from the training distribution and degrades classification accuracy.
- **Solution:** Add an LLM normalization step in the webhook between voice extraction and model inference:
  1. Raw conversational chief complaint comes in from Vapi
  2. A local LLM (Ollama + Llama 3.2 8B) extracts and rewrites it as a terse clinical phrase
  3. The normalized phrase is passed to BioBERT → PCA → v1.0.2-b for classification
- **Recommended stack:** `Ollama` (local, free, offline-capable) + `LangGraph` (manages conversation state across turns as a state machine)
- **Free cloud fallback:** Groq API (runs Llama 3.2, generous free tier, very fast) — swap in if demo machine is low on VRAM
- This keeps concerns cleanly separated: LLM handles language normalization, CatBoost handles clinical classification

**4. LangGraph RAG Integration (Hospital Protocols)**
- Enhance the webhook to utilize a LangGraph state machine.
- If the patient says something that triggers a specific emergency protocol (e.g., "bitten by a dog"), the LangChain agent intercepts, vector-searches a local hospital protocols database, and injects the specific follow-up questions the AI must ask next.

**4. Multilingual Capabilities**
- Enable automatic language detection within the Vapi configuration, allowing the bot to natively converse in Spanish/Hindi while strictly translating the resulting JSON payload back into English for our CatBoost model.

**5. CI/CD for Prompts**
- Move the system prompt and tool definitions out of the Vapi web dashboard and into our GitHub repository. Deploy them using REST API calls during backend startup to ensure version control of our prompt logic.
